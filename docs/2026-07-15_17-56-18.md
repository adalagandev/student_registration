# Agent gap analysis — two agents to add

**Generated:** 2026-07-15 17:56:18 CEDT

## Current agent roster

| Agent | Domain it owns |
|---|---|
| `ticket-warden` | Ticket-driven workflow — SR-`<n>` tickets, branch-per-ticket, commit prefixes (product-owner paperwork) |
| `controller-warden` | API boundary layer — controllers/routers, HTTP ↔ domain translation, status codes, DTOs at the edge |
| `service-architect` | Business logic — service classes, validation, transactions, DTO/domain mapping |
| `exception-warden` | Error handling — try/catch, exception mapping, `{"error": "..."}` bodies, status-code mapping |
| `test-guardian` | Tests — unit + integration/MockMvc suites, regression coverage |

## Where the coverage stops

The five existing agents form a complete **backend request spine**:
`ticket → controller → service → exception → test`. Two layers of the actual
codebase have no owner:

- **The React frontend** — the stated real purpose of this project
  (per `CLAUDE.md`) — has *no* agent. Every warden is backend-only.
- **The persistence layer** — `controller-warden` explicitly hands "database
  access, schemas, or repositories" to nobody. Entities, repositories, and the
  schema-migration path are ungoverned, despite `CLAUDE.md` calling out several
  fidelity-critical details there.

---

## Proposed agent 1 — `frontend-warden`

**Owns:** the React frontend (`frontend/src/`) — components, state, `api.js`,
styling tokens.

**Why it's needed**

`CLAUDE.md` states the project's *real purpose is to refresh React skills*, yet
none of the five agents touches `frontend/`. The frontend carries strict,
easy-to-break conventions that currently have no automated guardian:

- **`api.js` is the only place that talks to the backend.** Components must never
  call `fetch` directly — new endpoints get a function added in `api.js`.
- **State updates are immutable** (spread / `.map` / `.filter`, never mutation)
  so React re-renders; `App.jsx` owns all state and passes data + callbacks down.
- **Heavy teaching-comment density** — code doubles as React study material and
  documents JS/React keywords inline; edits must match that density, not strip it.
- **Stable, descriptive `id` attributes** on essentially every rendered element
  (`tab-register`, `student-row-{id}`, `edit-program-submit-btn`), following the
  `block-element` / interpolated-`{id}` naming so the set stays unique.
- **Theme via CSS custom properties** in `styles.css` — re-theme by changing
  `:root` tokens, never by scattering colors.
- **Program-change multipart** deliberately sets *no* `Content-Type` header so the
  browser adds the multipart boundary — a classic regression trap.

**Negative scope:** does not design backend endpoints, services, error bodies, or
tests — coordinates with `controller-warden` on the API contract and defers test
authoring to `test-guardian`.

**Suggested trigger:** PROACTIVELY on any change under `frontend/src/` — new or
edited components, state flow, `api.js` calls, or styling. MUST BE USED before
merging any frontend change.

## Proposed agent 2 — `persistence-warden`

**Owns:** the data layer — JPA entities, Spring Data repositories, and the
schema-migration path (`entity/`, `repository/`, `application.properties` DB config).

**Why it's needed**

`controller-warden`'s negative scope says it does *not* "design database access,
schemas, or repositories" — and no other agent picked that up. Meanwhile
`CLAUDE.md` is dense with data-layer fidelity rules that a specialist should guard:

- **Snake_case columns mapped via `@Column(name=...)`** on entities that back the
  pre-existing SQLite tables (`students`, `documents`) — the mapping must stay
  faithful to the legacy Python schema.
- **`StudentDto.from()` is the camelCase boundary** (the `row_to_dict()`
  equivalent) — the single conversion point between storage and wire shapes.
- **Schema migration is Hibernate `ddl-auto=update`** — a new field means editing
  the entity + DTO and letting Hibernate add the column; the agent guards against
  destructive or drifting schema changes.
- **Two tables with an FK** (documents → student), where a `documents` row records
  *both* the stored on-disk name and the original display name.
- **DB + upload paths come from `application.properties`** (`../backend/students.db`,
  `backend/uploads/<student_id>/`) and are shared 1:1 with the legacy Flask app —
  changes here risk breaking cross-backend parity.

**Negative scope:** does not put queries in controllers, own business rules
(defers to `service-architect`), or write tests (defers to `test-guardian`). It
guards the shape and integrity of storage, not what callers do with it.

**Suggested trigger:** PROACTIVELY when adding/changing an `@Entity`, a repository
interface, a column mapping, or DB configuration. MUST BE USED before merging any
schema or repository change.

## Summary

| New agent | Fills the gap of | Highest-value guarantee |
|---|---|---|
| `frontend-warden` | React frontend (the project's stated purpose) | `api.js`-only fetches, immutable state, teaching comments, stable `id`s |
| `persistence-warden` | Entities / repositories / schema migration | Faithful snake_case mapping, safe `ddl-auto` migrations, cross-backend parity |

Together these close the loop from **UI → API → service → data**, so every layer
of the stack has a specialist owner.
